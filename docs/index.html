The Problem

Multi-agent coding is chaos
without coordination

🔀

Context Desync

Agent A edits files that Agent B has cached. Neither knows the other's changes exist.

👻

Status Ambiguity

Both agents think they're the one working. File conflicts and duplicated effort follow.

🔍

No Audit Trail

Three handoffs ago, something broke. Good luck figuring out which agent did what.

🤷

Review Confusion

Who reviews what? When? How does the reviewer know what changed and why?

How It Works

One file. One protocol.
Any agent.

1

Claim

An agent claims a task via btrain handoff claim. Status → in-progress.

→
2

Work

The owner does the work. The reviewer waits. Locked files prevent conflicts.

→
3

Review

Owner runs btrain handoff update only after real work and reviewer context are ready. Status → needs-review. Reviewer acts.

→
4

Resolve

Reviewer runs btrain handoff resolve. Status → resolved. Next task.

The entire state lives in .claude/collab/HANDOFF_A.md plus any additional lane files — all git-tracked markdown. No database. No external service. Every transition is a CLI command.

Concurrent Lanes

Both agents work at once

Lane count comes from [agents].active and [lanes].per_agent. Each lane gets its own HANDOFF_<LANE>.md file, and file locks in .btrain/locks.json prevent collisions.

$ btrain handoff claim --lane a --task "Auth" \
  --owner Claude --reviewer Codex --files "src/auth/"

$ btrain handoff claim --lane b --task "Scoring" \
  --owner Codex --reviewer Claude --files "src/scoring/"

$ btrain locks
Active locks (2):
  lane a: src/auth/ (Claude)
  lane b: src/scoring/ (Codex)

Quick Start

Running in 60 seconds

Install
git clone https://github.com/codeslp/btrain.git
cd btrain && npm link
Bootstrap a repo
btrain init /path/to/your/repo --hooks \
  --agent "Claude" \
  --agent "GPT-5 Codex"
Claim a task
btrain handoff claim \
  --task "Implement auth middleware" \
  --owner "Claude" \
  --reviewer "GPT-5 Codex"
Type bth in each agent chat
# Claude sees: "in-progress, you're the active agent → do the work"
# Codex sees: "needs-review, you're the reviewer → review it"
# That's it. They follow the protocol.
Enable concurrent lanes (optional)
# Increase lane count per collaborator
btrain agents set --repo . \
  --agent "Claude" \
  --agent "GPT-5 Codex" \
  --lanes-per-agent 3

# Both agents claim separate lanes with file locks
btrain handoff claim --lane a --task "Auth" \
  --owner Claude --reviewer Codex --files "src/auth/"

btrain handoff claim --lane b --task "Scoring" \
  --owner Codex --reviewer Claude --files "src/scoring/"

🚧 Pre-Commit Guard

Run btrain init . --hooks to install a git hook that blocks commits when any tracked handoff shows needs-review. In multi-lane mode, it also blocks commits touching files locked by another lane.

Recommended Pairing

btrain + Spec Kit

btrain was built and spec'd using Spec Kit, a spec-driven planning tool for AI coding projects. We recommend using them together.

Spec Kit handles planning — constitution, specs, plans, tasks. btrain handles execution — who works on what, when, and how the handoff flows.

Future integration Planned

The obvious next step: btrain handoff claim --from-spec specs/001-feature/tasks.md — auto-create a handoff from the next unchecked task in a Spec Kit tasks file. This would close the loop between planning and execution, letting agents self-assign work from the spec backlog.

/speckit.specify — Create the feature spec

Transparency

What we tried: btrain loop

We built btrain loop to automate the bth relay — watch the active handoff file, dispatch the right agent CLI, repeat until resolved. The implementation works. The remaining limitation is what each agent CLI allows in practice.

Why it doesn't fully work in practice

Both Claude Code and Codex CLI have sandbox models that block the nested subprocess calls that btrain loop needs:

Codex (--full-auto)

Its sandbox uses sandbox-exec which blocks child processes from running btrain.
sandbox_apply: Operation not permitted

Claude Code (-p)

Requires --permission-mode bypassPermissions which is unsafe for production.

Antigravity

IDE-embedded with no CLI — uses notify mode (loop waits for manual action).

Current recommendation

Use bth manually in each agent's chat window. The loop infrastructure exists and will become practical as agent CLIs improve their headless orchestration support. btrain loop --dry-run still works for testing your runner configuration.
